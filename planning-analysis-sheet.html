Section 1Website Goal

The goal of this website is to establish a professional online presence for Niamh St John Lynch as a PhD Researcher, regulatory scientist, and international standardisation expert, so that academic collaborators, industry clients, and standards bodies can quickly understand her research, publications, consulting services, and training offerings — and contact her directly.

Desired result: Visitors leave with a clear understanding of the site owner's expertise in AI-enabled medical device regulation (EU MDR, EU AI Act, IEC TC62), her publication record, and the services she offers — with a frictionless route to make contact.

Target Audience

The site serves three overlapping professional audiences:

  • Academic collaborators and research supervisors — seeking a clear record of doctoral research activity, publications, and conference output. They need fast access to the full publication list and institutional affiliations.
  • Industry clients and regulatory professionals — medical device manufacturers, regulatory affairs teams, and quality managers seeking specialist consulting or training on EU MDR, the EU AI Act, and related standards. They need to understand the site owner's credentials and services quickly, and have a straightforward route to make an enquiry.
  • Standards bodies and national delegations — IEC, CEN-CENELEC, and NSAI peers seeking professional context on the site owner's standardisation roles and expert committee contributions. They need the Standards page to be authoritative and detailed.

The design — restrained colour palette, serif academic typography, concise professional copy — is calibrated for this audience: senior professionals in regulated industries who value credibility and substance over visual novelty. The site must work equally on a mobile device (checked at a conference) and on a desktop (used during a formal review).

Section 4Visual Design — Colour & Typography

Design Rationale

The visual design reflects the professional and academic nature of the site — doctoral research, international standardisation, and industry consulting in a regulated, high-stakes domain. The palette deliberately avoids bright or playful colours in favour of a restrained, authoritative aesthetic. The combination of deep navy with warm gold accents is a classic professional pairing used in academic publishing, legal, and financial sectors — conveying trust, precision, and seniority without appearing cold.

All colours are defined as CSS custom properties (variables) in :root in css/style.css. Changing a single variable value updates every instance across all pages — the design is systematic, not ad hoc.

Section 5Accessibility

The site targets WCAG 2.1 Level AA conformance throughout. Accessibility was treated as a first-class design requirement — not a retrofit — with semantic HTML, ARIA attributes, keyboard support, and colour contrast all addressed from the outset and audited during development.

Standard targeted: Web Content Accessibility Guidelines (WCAG) 2.1 Level AA

Reference: W3C WCAG 2.1 Quick Reference · WAI-ARIA Authoring Practices 1.1

Target audiences supported: Keyboard-only users · Screen reader users · Users with low vision · Users with vestibular / motion sensitivity

Accessibility Features Implemented

Feature Implementation WCAG Criterion
Skip link "Skip to main content" anchor link is the first focusable element in the DOM. Visible only on keyboard focus via :focus CSS, positioned off-screen at rest and slides in on focus. 2.4.1 Bypass Blocks (A)
Semantic HTML landmarks <nav>, <main>, and <footer> elements used with appropriate aria-label attributes, allowing screen reader users to jump directly to major page regions. 1.3.1 Info and Relationships (A) · 2.4.1 (A)
Page sections as regions Each SPA page section (<div>) carries role="region" and a descriptive aria-label. Inactive pages have aria-hidden="true" to suppress them from the accessibility tree; the active page has aria-hidden removed. 1.3.1 (A) · 4.1.2 Name, Role, Value (A)
Navigation toggle — ARIA state The hamburger button uses aria-expanded (toggled true/false), aria-controls="nav-links", and a dynamic aria-label ("Open navigation menu" / "Close navigation menu") so screen readers announce the menu state. 4.1.2 Name, Role, Value (A)
Active nav link indicator aria-current="page" is applied to the currently active navigation link and removed from all others on each page switch, giving screen reader users a clear indication of their location. 2.4.4 Link Purpose (A) · 2.4.8 Location (AA)
Keyboard navigation — nav links Navigation <a> elements styled as buttons use role="button" and tabindex="0". JavaScript keydown listeners fire the click action on Enter and Space, matching expected button keyboard behaviour. 2.1.1 Keyboard (A)
Escape key menu dismissal Pressing Escape closes the mobile navigation menu and returns focus to the toggle button, following the ARIA disclosure button pattern. 2.1.1 Keyboard (A)
Focus management on page switch When a page is shown, it receives tabindex="-1" and programmatic .focus(), moving keyboard focus to the top of the new page content. preventScroll: true is used to avoid jarring scroll behaviour. 2.4.3 Focus Order (A)
Visible focus indicators :focus-visible pseudo-class applies a 3px gold outline on all interactive elements. Form controls use an accent-colour outline. Default outline suppression (outline: none) is avoided. 2.4.7 Focus Visible (AA)
Colour contrast — body text All body text colours (#3a3a5c on white, #1a1a2e on cream) exceed 7:1 contrast ratio (AAA). 1.4.3 Contrast Minimum (AA)
Colour contrast — hero text Hero text colours were audited and corrected during development. .hero-desc changed from #a0a8bc (ratio ~3.2:1, failing) to #c8cfe0 (ratio 4.6:1, passing AA) on the dark #1a1a2e background. 1.4.3 Contrast Minimum (AA)
Form — live regions Contact form success message uses role="status" and aria-live="polite"; the error message uses role="alert" and aria-live="assertive". Both are announced automatically by screen readers without requiring a page reload. 4.1.3 Status Messages (AA)
Form — button loading state The submit button sets aria-busy="true" and updates its text to "Sending…" during form submission, informing assistive technology users that an action is in progress. 4.1.2 Name, Role, Value (A)
Form labels All form inputs have associated <label> elements linked via matching for / id attributes. No placeholder-only labelling is used. 1.3.1 (A) · 2.4.6 Headings and Labels (AA)
SVG figures — alternative text The inline SVG research diagram uses role="img" on its container with a full descriptive aria-label. Internal SVG elements are marked aria-hidden="true" to prevent screen readers from traversing individual shapes. 1.1.1 Non-text Content (A)
External links Links that open in a new tab (target="_blank") include rel="noopener" for security and, where needed, an aria-label indicating the link opens in a new tab (e.g. DkIT STÓR links). 2.4.4 Link Purpose (A)
Decorative elements hidden Purely decorative elements (section label lines, stat dividers, icon emojis) carry aria-hidden="true" to remove them from the accessibility tree. 1.1.1 Non-text Content (A)
Reduced motion The @media (prefers-reduced-motion: reduce) media query disables all CSS animations (fade-up page transitions) and sets scroll-behavior: auto, respecting users who have requested reduced motion at the OS level. 2.3.3 Animation from Interactions (AAA) · good practice
Responsive / zoom The <meta viewport> tag does not set maximum-scale or user-scalable=no, so users can zoom up to 400% without loss of content or functionality. 1.4.4 Resize Text (AA)
Table accessibility The training sessions table uses <thead>, <th scope="col">, and is wrapped in a scrollable role="region" container with tabindex="0" and aria-label so keyboard users can scroll horizontally. 1.3.1 (A) · 2.1.1 (A)

Section 6Current Software Development Approaches & AI-Assisted Workflow

Software Development Approaches

The project followed established modern web development practices throughout:

  • Version control with Git and GitHub — every change committed with descriptive messages, full history preserved
  • Iterative development — pages built and refined incrementally (home → research → standards → consulting → training → contact), with each commit representing a working state
  • Mobile-first responsive design — CSS Grid auto-fit/minmax and @media breakpoints ensure the site works at all screen sizes without a CSS framework
  • Separation of concerns — HTML structure in index.html, styles in css/style.css, behaviour in js/script.js (later consolidated inline for SPA architecture)
  • Accessibility-first — semantic HTML5 landmarks, ARIA roles/labels, keyboard navigation, skip link, live regions, and prefers-reduced-motion media query applied from the outset
  • Progressive enhancement — the page is readable without JavaScript; JS only adds navigation switching behaviour on top of semantic HTML

Terry Morris Web Design Best Practices — How They Were Applied

The site was designed and built with reference to the recommended web design practices at terrymorris.net/bestpractices. The table below maps the key recommendations to specific implementation decisions made in this project.

Best Practice How It Was Applied in This Project
Know your audience Design, copy tone, and content depth are calibrated for regulatory professionals, academics, and standards experts — not general consumers. No jargon is avoided; instead, specialist terminology is used accurately because the audience expects it.
Provide good navigation A sticky navigation bar is present on all pages with clearly labelled links. The active page is highlighted with aria-current="page" and a gold background. A skip link allows keyboard users to bypass navigation. On mobile, a hamburger menu provides full navigation access.
Use white space effectively Generous padding on all sections (5rem 2rem), cards, and publication items prevents the page feeling cluttered despite the volume of information. The alternating cream/white section backgrounds create visual rhythm without requiring decorative elements.
Limit the number of fonts Exactly two fonts are used throughout — Cormorant Garamond (headings) and DM Sans (body). No other font families are introduced. Hierarchy is achieved through size, weight, and letter-spacing within these two families.
Use colour consistently Ten colours are defined as CSS custom properties in :root and used consistently across all six pages. The same gold accent, navy background, and blue interactive colour appear in the same roles everywhere on the site.
Design for scannability Card grids, publication lists with gold left borders, section labels in small caps, and bold eyebrow text allow visitors to scan the page and extract key information without reading every word. Statistics are displayed in a prominent strip with large numerals.
Ensure fast loading No frameworks, no large JavaScript bundles, no hero images. The only external requests are two Google Font families (loaded asynchronously) and the Formspree API (only called on form submit). All diagrams are inline SVG — zero extra HTTP requests.
Design for accessibility WCAG 2.1 Level AA is the design target. Skip links, semantic HTML, ARIA labels, keyboard navigation, colour contrast verification, and reduced motion support are all implemented. See Section 5 for the full accessibility feature list.
Design for multiple browsers and devices The site uses standard HTML5, CSS3, and vanilla JavaScript with no proprietary APIs. CSS Grid and Flexbox are supported by all modern browsers. The mobile-first responsive layout is tested at 375px (iPhone SE), 768px (tablet), and 1440px (desktop).
Provide clear calls to action Each page section ends with prominent CTA buttons (gold filled for primary, outlined for secondary). The hero on every main page includes a direct route to the Contact page. The contact form is always reachable within two clicks from any page.

Section 7Responsive Design

The site is built mobile-first — base styles target small screens and @media breakpoints progressively enhance the layout for larger viewports. No CSS framework (Bootstrap, Tailwind, etc.) is used; all responsive behaviour is achieved with native CSS Grid, Flexbox, and two media query breakpoints.

Approach: Mobile-first — base styles for small screens, enhanced at breakpoints

Techniques: CSS Grid auto-fit / minmax(), Flexbox flex-wrap, clamp() fluid typography, @media breakpoints

Breakpoints: 768px (tablet threshold) · 1100px (desktop / max-width container)

Framework used: None — all layout is handwritten CSS

Breakpoints & Layout Changes

Viewport Breakpoint Layout Behaviour
Mobile < 768px Single column throughout. Hero collapses to one column; affiliations card is hidden (display: none). Horizontal navigation links are hidden; hamburger menu button appears. Card grids wrap to one column. Contact grid stacks vertically. Standards items stack (badge above text). Stats strip items wrap to two per row with dividers hidden.
Tablet ≥ 768px Two-column card grids active (minmax(300px, 1fr)). Hero two-column layout restores. Contact side-by-side layout active. Standards badge and text return to horizontal grid. Hamburger menu hidden; horizontal nav links shown. Affiliations card visible.
Desktop ≥ 1100px (max-width) Three-column card grids active (minmax(280px, 1fr)). All content containers capped at max-width: 1100px and centred with margin: 0 auto. Hero grid uses fixed grid-template-columns: 1fr 320px. No further breakpoint beyond this — layout scales within the container.

Component-Level Responsive Behaviour

Component Mobile (< 768px) Tablet / Desktop (≥ 768px)
Navigation Hamburger button (☰) replaces nav links. Tapping opens a full-width dropdown. Menu closes on link tap, outside click, or Escape key. Horizontal nav links visible in a flex row. Hamburger button hidden (display: none).
Hero section Single column. Headline and body text stack above CTAs. Affiliations card hidden to reduce scroll depth on small screens. Two-column grid: text content left, affiliations card right (1fr 320px).
Stats strip Items wrap to two per row. Vertical dividers hidden (display: none). Reduced padding per item. All 5 stats in a single horizontal row with gold dividers between each.
Card grids Single column — each card takes full width. 2-column grid on tablet (minmax(300px, 1fr)); 3-column on desktop (minmax(280px, 1fr)) via auto-fit.
Standards items Badge stacks above text content (grid-template-columns: 1fr). Badge and text side by side (grid-template-columns: auto 1fr).
Contact layout Info column and form stack vertically with 2.5rem gap. Side-by-side two-column grid (1fr 1fr) with 4rem gap.
Training table Horizontally scrollable — wraps in overflow-x: auto container with role="region" and tabindex="0" for keyboard scrollability. Table displays at full width without scrolling.
Typography clamp() on headings scales font size fluidly: e.g. clamp(2.8rem, 5vw, 4.2rem) for the hero H1 and clamp(1.8rem, 3vw, 2.6rem) for section titles. Same clamp() values scale up smoothly — no abrupt size jump at breakpoints.
SVG research figure SVG uses width: 100%; height: auto — scales proportionally to container width. Capped at max-width: 100% within the section container.

Mobile Navigation — Detail

On viewports below 768px the horizontal nav links are hidden with CSS and a hamburger icon button is shown. The full behaviour:

  • Tapping the ☰ button adds the .open class to the #nav-links list, which switches it from display: none to display: flex; flex-direction: column — a full-width dropdown below the nav bar
  • Each nav link takes 100% width with left padding and a transparent left border that turns gold when active
  • The menu closes automatically when a nav link is tapped (via the showPage() function calling closeMenu()), when the user clicks anywhere outside the nav, or when the Escape key is pressed
  • aria-expanded on the toggle button is updated in real time so screen readers always know whether the menu is open or closed

CSS Techniques — Summary

Technique Where Used Purpose
CSS Grid auto-fit / minmax() All card grids (.card-grid-2, .card-grid-3) Cards automatically reflow from 3 → 2 → 1 column as the viewport narrows, with no explicit breakpoint needed per grid
CSS Grid fixed columns Hero (1fr 320px), contact (1fr 1fr), standards items (auto 1fr) Fixed two-column layouts that collapse to single column at the 768px breakpoint
Flexbox + flex-wrap Nav links, hero CTAs, stats strip, affiliation badges Items wrap naturally when space is insufficient, without overflow
clamp(min, preferred, max) Hero H1, section titles Fluid font sizes that scale smoothly between minimum and maximum values based on viewport width — no font-size breakpoints needed
@media (max-width: 768px) Nav toggle, hero grid, contact grid, standards items, stats strip Single breakpoint handles all layout switches from multi-column to single-column
@media (max-width: 600px) Stats strip only Additional breakpoint hides stat dividers and reduces stat item padding on very small screens
max-width: 1100px; margin: 0 auto All .section-inner, .nav-inner, .hero-inner, .stats-inner Constrains content width on large monitors and centres it, preventing overly wide line lengths
